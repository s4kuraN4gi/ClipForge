### 新規発見事項

#### CRITICAL

**S-17: Safety Cap のレースコンディション（Pro プラン）**

- **ファイル**: `src/lib/subscription.ts` + `src/app/api/generate/route.ts`
- **内容**: `checkVideoLimit()` と `incrementVideoCount()` が非アトミック。同時リクエストで safety cap (50本) を超過する可能性がある
- **影響**: Pro ユーザーが同時に複数タブから生成すると、50本の上限を突破できる
- **対策**: Free プランと同様に `try_increment_video_count` RPC を Pro にも適用し、safety cap をDB側で enforcement する

```sql
-- 推奨: Pro 用のアトミックインクリメント
CREATE OR REPLACE FUNCTION try_increment_video_count_pro(
  target_user_id UUID,
  safety_cap INT
) RETURNS BOOLEAN AS $$
DECLARE success BOOLEAN;
BEGIN
  UPDATE subscriptions
  SET monthly_video_count = monthly_video_count + 1, updated_at = now()
  WHERE user_id = target_user_id
    AND monthly_video_count < safety_cap
  RETURNING TRUE INTO success;
  RETURN COALESCE(success, FALSE);
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;
```

**S-18: メーター使用量報告の静かな失敗**

- **ファイル**: `src/app/api/generate/route.ts` (L233-241)
- **内容**: `reportMeteredUsage` が失敗しても `console.error` のみで生成は続行される。Stripe に課金情報が届かず、無課金で動画が生成される
- **影響**: Stripe API 障害時に追加動画が無料で生成され続ける。事業者側の損失
- **対策**:
  1. 失敗時にDB に `metered_report_pending` フラグを立てる
  2. バックグラウンドジョブ（cron）で未報告分をリトライ
  3. または、usage 報告成功をゲートにして、失敗時は生成をブロック

#### HIGH

**S-19: サブスクリプション past_due 状態の未チェック**

- **ファイル**: `src/lib/subscription.ts`
- **内容**: `checkVideoLimit` で subscription の `status` を確認していない。`past_due`（支払い遅延）のユーザーも通常通り生成できる
- **対策**: `status !== "active"` のサブスクリプションは `allowed: false` にする

**S-20: storagePaths の未検証**

- **ファイル**: `src/app/api/generate/route.ts` (L185-199)
- **内容**: `body.storagePaths` がそのまま DB に保存される。パストラバーサルや任意パス挿入のリスク
- **対策**: `storagePaths` が `{userId}/` プレフィックスで始まることを検証

**S-21: Webhook 冪等性のレースコンディション**

- **ファイル**: `src/app/api/stripe/webhook/route.ts`
- **内容**: `processed_webhook_events` テーブルへの INSERT と処理の間にレースがある。同じイベントが同時に2回届いた場合、両方とも処理される可能性
- **対策**: INSERT に `ON CONFLICT DO NOTHING` を使い、affected rows = 0 なら早期リターン

**S-22: confirmExtra バイパス**

- **ファイル**: `src/app/api/generate/route.ts`
- **内容**: `confirmExtra: true` をリクエストに最初から付ければ、確認ダイアログなしで追加課金動画を生成できる。技術的にはAPIレベルで防止不可能だが、サーバー側でも「現在metered圏内である」ことを再検証すべき
- **対策**: `confirmExtra` が送られた場合でも、`isMetered` が false なら無視する（現在は実質問題ないが、防御的プログラミングとして）

#### MEDIUM

**S-23: Billing Meter Events のリトライなし**

- **ファイル**: `src/lib/subscription.ts` (L126-134)
- **内容**: `meterEvents.create` が失敗した場合のリトライロジックがない。for ループ内で途中失敗すると、quantity の一部しか報告されない
- **対策**: 個別の失敗をキャッチし、未報告分をDB に記録してリトライ

**S-24: エラーメッセージのローカライゼーション漏れ**

- **ファイル**: 各APIルート
- **内容**: エラーメッセージが日本語固定。国際化を想定する場合はエラーコードベースに変更が必要
- **影響**: 現時点では日本市場のみなので LOW だが、今後の拡大時に問題

### セキュリティ改善後の想定評価: B+

S-17 と S-18 を修正すれば B+ に到達可能。全 HIGH も修正で A- が視野に入る。

### 新規発見事項

#### CRITICAL

**C-1: ROI（投資対効果）の証明が不在**

- **内容**: 「¥980/月で動画が作れる」だけでは弱い。ターゲットユーザー（月売上5万円以下の作家）に対して「この動画で売上がどれだけ上がるか」のケーススタディや実績データがない
- **影響**: 無料体験後の有料転換率が低くなる
- **対策**:
  1. β版ユーザーの売上改善データ収集
  2. LP に「動画を使った作家の売上変化」セクション追加
  3. 最低でもモックでも良いので Before/After の売上比較を掲載

**C-2: 12テンプレート中、サンプル動画は3本のみ**

- **内容**: テンプレートは12種に拡充したが、実際のサンプル動画が用意されているのは3種のみ（旧テンプレート分）。ユーザーは仕上がりを見ずにテンプレートを選ぶ必要がある
- **影響**: テンプレート選択時のUXが悪く、期待と結果のミスマッチが起きやすい
- **対策**: 全12テンプレートのサンプル動画を制作し、LP とテンプレート選択画面に設置

**C-3: 15秒動画のAIコストが未検証**

- **内容**: 5秒動画のコスト（約¥45）は把握済みだが、15秒動画のコストが不明。15秒が3倍コスト（¥135）だと追加¥200/本でも利益率が大幅に下がる
- **影響**: 15秒動画が多用された場合の収益性が不透明
- **対策**:
  1. WaveSpeed の 15秒動画の実コストを計測
  2. コストに応じて duration 別の追加料金設定を検討
  3. または Pro の含む分は5秒のみ、15秒は常に追加料金にする

#### HIGH

**H-1: 年額プランがない**

- **内容**: SaaS の一般的な慣行として年額プラン（割引付き）がない。年額は解約率を下げ、キャッシュフローを安定させる
- **対策**: 年額 ¥9,800（2ヶ月分無料）を追加。ただしユーザー数が少ない初期段階では月額のみでも可

**H-2: 追加料金ダイアログに累計表示がない**

- **内容**: 確認ダイアログで「¥200で生成する」と表示されるが、今月の累計追加料金が見えない。ユーザーは月末にいくら請求されるか把握できない
- **対策**: ダイアログに「今月の追加生成: X本（累計 ¥Y）→ この生成後: X+1本（累計 ¥Z）」を表示

**H-3: ダッシュボードに請求予定額がない**

- **内容**: subscription-card に含む分と追加本数は表示されるが、「今月の請求予定額」がない
- **対策**: 「¥980 + 追加 X本 × ¥200 = ¥Y（税別）」を表示

**H-4: FAQ に「Seedance」の記載が残っている**

- **内容**: LP の FAQ で「Seedance（Bytedance社）のAIエンジン」と記載されているが、実際は WaveSpeed AI (Wan 2.1) を使用
- **影響**: ユーザーに誤った情報を提供。信頼性の問題
- **対策**: FAQ を WaveSpeed AI (Wan 2.1) に修正

#### MEDIUM

**M-6: 解約フローのフリクション不足**

- **内容**: Stripe Portal からワンクリックで解約可能。解約理由の収集や、代替プランの提案がない
- **対策**: 解約時にアンケート + 「一時停止」オプション提供

**M-7: アクティベーションメトリクスの不在**

- **内容**: ユーザーがサインアップしてから最初の動画生成までのファネルを計測していない
- **対策**: PostHog/Mixpanel 等でアクティベーション率を計測（課金ユーザー出てから）

**M-8: リファラルプログラムがない**

- **内容**: 口コミを加速する仕組みがない。作家コミュニティは口コミが有効なチャネル
- **対策**: 「友達紹介で1本無料」等のリファラルプログラムを検討
